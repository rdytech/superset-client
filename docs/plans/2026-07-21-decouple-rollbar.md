# Decouple Rollbar & Introduce Configurable Logger — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove all Rollbar coupling from `superset-client` (a public gem) and replace the hardcoded logger with a `Superset.configure { |c| c.logger = ... }` API so downstream apps can inject their own logger. The gem must be framework-neutral — it cannot presume the consumer is a Rails app or any other particular framework. Any object that responds to `#info(msg)` and `#error(msg)` should work as an injected logger.

**Architecture:** Delete the gem's per-dataset `begin/rescue/end` in `WarmUpCache` entirely — exceptions propagate to the caller, whose own rescue (e.g., Ready Apprentice's outer Rollbar-reporting rescue) already knows how to react. Introduce `Superset::Configuration` (attr_accessor for `logger`) and module-level `Superset.configure` / `Superset.configuration` / `Superset.logger` methods for the *rest* of the gem's logging surface (`Request#response`, `DuplicateDashboard`, etc.). Route the `Superset::Logger` wrapper class through the configured logger, falling back to `::Logger.new("log/superset-client.log")` so existing consumers see no behavior change. Delete the stale Rollbar spec, the `require "rollbar"` in spec_helper, the Rollbar dev dependency in the gemspec, and the dead `DuplicateDashboardLogger` module.

**Tech Stack:** Ruby ≥ 3, RSpec, Bundler, standard library `::Logger`.

## Task 1: Delete the per-dataset rescue in WarmUpCache; let exceptions propagate

**Files:**
- Modify: `lib/superset/dashboard/warm_up_cache.rb:15-25`
- Test: `spec/superset/dashboard/warm_up_cache_spec.rb`

**Rationale:** The gem's per-dataset `begin/rescue/end` was reaching into consumers' namespace via `Rollbar.error(...) if defined?(Rollbar)` — a coupling violation for a public gem. The rescue also *hid* dataset-level failures from consumers' own outer rescues (e.g., Ready Apprentice wraps `Superset::Dashboard::WarmUpCache.new(...).perform` in its own `rescue => e; Rollbar.error(...); end`, which never sees per-dataset failures because they're swallowed inside the gem). Deleting the inner rescue lets consumers decide how to react — Rollbar-report, log, ignore, retry. Per-dataset resilience within a single dashboard run is lost (a mid-loop failure aborts remaining datasets for that dashboard), which is acceptable for a cache warmer (things rebuild on demand, and next scheduled run reattempts).

```ruby
      end
```

```ruby
        Superset::Dashboard::Datasets::List.new(dashboard_id: dashboard_id).datasets_details.map { |dataset| dataset['database'].slice('name').merge(dataset.slice('datasource_name'))}
      end
    end
  end
end
```

The gem writes logs (dashboard duplication progress, API failures, etc.) through `Superset.logger`. By default it writes to `log/superset-client.log` relative to the working directory.

```
## Summary
Removes all Rollbar coupling from superset-client (a public gem). Deletes the per-dataset rescue in `WarmUpCache` so exceptions propagate to the caller. Introduces a `Superset.configure { |c| c.logger = ... }` API so downstream apps can inject their own logger for the rest of the gem's logging surface.

## Why
Two reasons:
1. **Library correctness.** The gem was reaching into consumers' namespace via `Rollbar.error(...) if defined?(Rollbar)` — a coupling violation for a public gem. It was also *hiding* per-dataset failures from consumers' own rescues: Ready Apprentice wraps `Superset::Dashboard::WarmUpCache.new(...).perform` in its own `rescue => e; Rollbar.error(...); end`, but never sees per-dataset failures because they were swallowed inside the gem. Deleting the inner rescue lets consumers own the response to failures.
2. **Signal-to-noise (why we noticed).** Ready Apprentice was receiving ~6 gem-side Rollbar items/day from `WarmUpCache`. Its outer rescue will keep any dashboard-level failures on RA's radar via its existing Rollbar call; the removed inner reports were the redundant, gem-side ones. See https://app.rollbar.com/a/JobReady/fix/item/Active/40164#occurrences.

## Changes
- **`WarmUpCache`**: delete the per-dataset `begin/rescue/end`. Exceptions now propagate. Consumers' own rescue decides how to react (Rollbar, log, ignore).
- **`Superset::Configuration` + module accessors**: `Superset.configure { |c| c.logger = ... }`, `Superset.configuration`, `Superset.logger`, `Superset.reset_configuration!`. Default logger unchanged (`log/superset-client.log`).
- **`Superset::Logger` shim**: now delegates `#info` and `#error` to `Superset.logger`. All 5 existing callers (`Request`, `Dataset::WarmUpCache`, `BulkDeleteCascade`, `Dashboard::Export`, `DuplicateDashboard`) keep working with no changes.
- **Dead code**: delete unused `DuplicateDashboardLogger` module (defined but never included; had the same `defined?(Rails)` reach-in anti-pattern).

- **Dev dep**: drop `rollbar` from `superset.gemspec` and `require "rollbar"` from `spec_helper.rb`.

## Behavior change to be aware of
Consumers that call `Superset::Dashboard::WarmUpCache.new(...).perform` will now see raised exceptions where before they saw silent, per-dataset log entries. Ready Apprentice already handles this via its outer `rescue`. Any other consumer that relied on the gem's silent behavior should add their own rescue.

Per-dataset resilience within a single dashboard run is lost — a mid-loop failure aborts the remaining datasets in that dashboard until the next scheduled run. Acceptable for a cache warmer (caches rebuild on demand).

## Backwards compatibility
Full for the logger API. If a consumer does not call `Superset.configure`, logs still go to `log/superset-client.log` as before. All 5 existing callers of `Superset::Logger.new` continue to work unchanged.

Not backwards-compatible for `WarmUpCache` — see "Behavior change" above.
```
